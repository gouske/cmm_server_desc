# 파트 1. 프로젝트 전체 분석 요약

> **프로젝트명**: CMM Server  
> **한 줄 요약**: 여러 명이 동시에 접속해 게임을 즐길 수 있도록 지원하는 멀티플레이어 게임 서버 시스템

---

## 어떤 프로젝트인가

추격자와 생존자로 나뉘어 싸우는 멀티플레이어 게임의 서버 시스템입니다.  
로그인부터 매칭, 실제 게임 진행, 결과 저장, 랭킹 집계까지 모든 서버 측 기능을 포함합니다.  
6개의 독립 서버와 여러 공용 라이브러리로 구성되어 있으며, 개발·테스트·운영 환경을 분리해 관리합니다.

---

## 기술 스택

| 분류 | 기술 | 하는 일 |
|------|------|---------|
| 개발 언어 | C# (.NET Core 3.1) | 서버 프로그램 작성 |
| 공용 라이브러리 | .NET Standard 2.1 | 여러 서버에서 함께 쓰는 코드 묶음 |
| 네트워크 | SuperSocket | 클라이언트 연결 수신·처리 |
| 데이터 전송 형식 | MessagePack | 데이터를 빠르고 작게 압축해 주고받음 |
| 설정 파일 처리 | Newtonsoft.Json | 설정값을 JSON으로 읽고 쓰기 |
| 로그 | NLog | 서버 상태와 오류를 파일로 기록 |
| 데이터베이스 | MySQL | 유저 계정, 게임 기록 등 영구 저장 |
| 빠른 임시 저장소 | Redis | 접속 중인 유저 정보, 매칭 상태 등 빠른 읽기/쓰기 |
| 보안 | AES-256 암호화 | 주고받는 데이터가 중간에 탈취·변조되지 않도록 보호 |
| 클라우드 | AWS ElastiCache | 운영 환경에서 Redis를 안정적으로 제공 |

```
├── cmm_bridge/         ← 서버 간 메시지 중계 서버 (포트 4314)

```

## 서버별 역할

| 서버 | 맡은 일 |
|------|---------|
| **cmm_server** | 로그인 확인, 캐릭터 정보 불러오기 |
| **cmm_lobby** | 로비 입장, 친구·클랜 관리, 채팅 |
| **cmm_match** | 게임 대기열 관리, 방 생성 |
| **cmm_bridge** | 서버들 사이 메시지 전달 (우편함 역할) |
| **cmm_game** | 실제 게임 진행, 스킬·맵·보상 처리 |
| **cmm_rank** | 랭킹 점수 계산, 시즌 관리 |

---

## 게임 한 판의 흐름

```
① 유저가 앱 실행
       ↓
② [cmm_server] 로그인 확인 → 캐릭터 데이터 로딩
       ↓
③ [cmm_lobby] 로비 입장 → 친구 목록, 채팅
       ↓  "게임 찾기" 버튼 클릭
④ [cmm_match] 비슷한 실력의 유저들을 모아 방 생성
       ↓
⑤ [cmm_bridge] 게임 서버로 연결 요청 중계
       ↓
⑥ [cmm_game] 게임 진행 (100ms마다 상태 갱신)
       ↓  게임 종료
⑦ [cmm_rank] 결과 반영 → 랭킹 점수 업데이트
       ↓
⑧ [Redis] 접속 정보·매칭 상태 빠른 저장
   [MySQL] 게임 기록·계정 정보 영구 저장
```

# 파트 2. 기술적 하이라이트

---

## 1. Redis 읽기 전용 / 쓰기 전용 연결 분리

**어떤 문제를 해결했나**
데이터를 읽는 요청과 저장하는 요청이 같은 연결을 함께 쓰면, 저장 요청이 많아질 때 읽기 요청도 덩달아 느려지는 병목 현상이 생깁니다.

**어떻게 해결했나**
Redis 연결을 읽기 전용 풀과 쓰기 전용 풀로 완전히 분리했습니다. 두 풀 모두 부하가 가장 적은 연결로 요청을 자동 분배(LeastLoaded 전략)하며, 연결 오류가 3회 누적되면 자동으로 재연결합니다.

**왜 중요한가**
읽기와 쓰기가 서로 방해하지 않아 동시에 수천 명이 접속해도 빠른 응답 속도를 유지할 수 있습니다.

---

## 2. 여러 패킷을 묶어서 한 번에 전송

**어떤 문제를 해결했나**
게임 중에는 아주 짧은 시간 안에 수십~수백 개의 작은 데이터 조각(패킷)이 발생합니다. 이것을 하나씩 따로 보내면 운영체제에 요청을 반복해야 해서 속도가 떨어집니다.

**어떻게 해결했나**
각 스레드(처리 흐름)가 전용 버퍼를 따로 갖고, 패킷들을 최대 1MB 크기로 모은 다음 단 한 번에 전송합니다. 수신 버퍼는 고정 배열을 재사용하고, ArraySegment 슬라이싱으로 불필요한 메모리 할당을 최소화하는 방식을 적용했습니다.

**왜 중요한가**
운영체제 전송 요청 횟수가 대폭 줄어들고, 스레드끼리 버퍼를 공유하지 않아 충돌도 없습니다.

---

## 3. 두 개의 작업 목록을 번갈아 사용하는 대기열

**어떤 문제를 해결했나**
새 작업을 추가하는 쪽과 작업을 꺼내 처리하는 쪽이 같은 목록에 동시에 접근하면 서로 기다려야 하는 상황이 생깁니다.

**어떻게 해결했나**
작업 목록을 두 개 만들어 번갈아 사용합니다. 한쪽에 작업을 넣는 동안 반대쪽 목록의 작업을 처리하고, 처리가 끝나면 역할을 바꿉니다. 목록 교체는 잠금 장치 없이 원자적으로 처리합니다.

**왜 중요한가**
서로 기다리는 시간이 없어 매칭 서버 등 바쁜 구간에서도 빠르게 메시지를 처리합니다.

---

## 4. 무거운 작업은 여럿이, 상태 변경은 혼자서

**어떤 문제를 해결했나**
게임 상태(점수, 위치 등)를 여러 처리 흐름이 동시에 바꾸면 데이터가 꼬일 수 있습니다. 하지만 하나씩 처리하면 CPU 여러 개를 낭비하게 됩니다.

**어떻게 해결했나**
작업을 두 단계로 나눴습니다.
- **1단계**: 여러 백그라운드 스레드가 CPU 집약 작업(패킷 해석 등)을 동시에 처리
- **2단계**: 메인 스레드 하나만 게임 상태를 변경

**왜 중요한가**
CPU 성능을 최대한 활용하면서도 데이터 일관성을 안전하게 보장합니다. 게임 서버에서 가장 이상적인 동시 처리 구조입니다.

---

## 5. 실행 시간 기준으로 정렬된 작업 예약 시스템

**어떤 문제를 해결했나**
스킬 쿨타임, 세션 만료, 매칭 제한 시간 등 수천 개의 예약된 작업을 정확한 순서대로 실행해야 합니다.

**어떻게 해결했나**
실행 시각을 기준으로 정렬되는 우선순위 목록(최소 힙)을 직접 구현했습니다. 가장 빨리 실행해야 할 작업을 O(log N)의 시간 복잡도로 꺼낼 수 있고, 아직 실행 시각이 안 된 작업은 즉시 건너뜁니다.

**왜 중요한가**
작업 수가 늘어나도 처리 속도가 안정적으로 유지되며, 게임 타이밍 이벤트가 정확한 순서로 동작합니다.

---

## 6. 필요할 때만 정리하는 수신 버퍼

**어떤 문제를 해결했나**
서버가 데이터를 받을 때마다 메모리를 새로 할당하면 메모리 쓰레기 수집(GC)이 자주 발생해 성능이 저하됩니다.

**어떻게 해결했나**
고정 크기 버퍼에 읽은 위치와 쓴 위치를 별도로 관리합니다. 데이터를 읽을 때는 복사하지 않고 "시작~끝" 위치만 알려줍니다. 버퍼가 꽉 찰 때만 한 번에 정리합니다.

**왜 중요한가**
접속한 모든 세션에 공통 적용되므로, 접속자 수가 많을수록 절약 효과가 커집니다.

---

## 7. 잠금 없이 상태를 바꾸는 원자적 제어

**어떤 문제를 해결했나**
여러 스레드가 동시에 같은 값을 바꾸려 할 때 잠금(lock)을 걸면, 기다리는 스레드가 쌓여 성능이 저하됩니다.

**어떻게 해결했나**
CPU 명령어 수준에서 지원하는 비교-교체(Compare-And-Swap) 연산을 활용했습니다. "현재 값이 0이면 1로 바꿔라"를 잠금 없이 원자적으로 처리합니다.

**왜 중요한가**
모든 패킷이 통과하는 핵심 경로에서 잠금을 제거해 병목을 방지합니다.

---

## 8. 실력 등급별 매칭 + 파티 분리 방지

**어떤 문제를 해결했나**
실력 차이가 큰 유저가 같은 게임에 배정되거나, 함께 게임 찾기를 한 파티원이 다른 방으로 분리되는 문제가 있을 수 있습니다.

**어떻게 해결했나**
등급별로 독립적인 방 목록을 관리합니다. 듀오 파티는 두 명을 반드시 같은 방에 함께 배정합니다. 매칭 스레드가 사이클마다 대기 중인 유저들을 일괄 배정합니다.

**왜 중요한가**
공정한 게임 환경을 보장하고, 파티 플레이 경험이 깨지지 않습니다.

---

## 9. 중앙 우편함 역할의 메시지 라우터 (Bridge 서버)

**어떤 문제를 해결했나**
6개 서버가 서로 직접 연결되면 연결 수가 기하급수적으로 늘어나고, 서버 주소가 바뀔 때마다 여러 곳의 코드를 수정해야 합니다.

**어떻게 해결했나**
Bridge 서버를 중앙 허브로 두고 모든 서버간 메시지를 여기서 중계합니다. 각 서버는 Bridge 주소만 알면 다른 서버와 통신할 수 있습니다.

**왜 중요한가**
서버를 추가하거나 주소가 바뀌어도 Bridge 한 곳만 수정하면 됩니다. 서버 간 결합도가 낮아져 유지보수가 편합니다.

---

## 10. 전 구간 AES-256 패킷 암호화

**어떤 문제를 해결했나**
데이터를 암호화하지 않으면 네트워크를 가로채 게임 패킷을 분석하거나 변조하는 치트 도구를 만들 수 있습니다.

**어떻게 해결했나**
클라이언트와 서버 사이의 모든 데이터를 AES-256 방식으로 암호화합니다. ECB 방식(블록이 반복되면 패턴이 보임)은 피하고 CBC 방식을 선택했습니다. 암호화 키는 환경(개발/테스트/운영)별로 분리 관리합니다.

**왜 중요한가**
패킷을 가로채도 내용을 알 수 없고 변조도 불가능해집니다. 환경별 키 분리로 운영 키가 개발 과정에서 노출되는 위험도 차단합니다.

# 파트 3. 경력 기술서 — 핵심 업무 및 성과

> **프로젝트**: 멀티플레이어 게임 서버 시스템  
> **기술 스택**: C# · .NET Core 3.1 · SuperSocket · MessagePack · MySQL · Redis · AWS  
> **역할**: 서버 엔진 설계 및 구현, 공용 라이브러리 개발, 게임 서버 시스템 구축  

---

## 1. 네트워크 전송 성능 개선

### 1-1. 패킷 묶음 전송 시스템 설계 및 구현

**[상황]** 게임 특성상 짧은 시간 안에 수십~수백 개의 작은 데이터 조각이 발생하는데, 이를 하나씩 따로 전송하면 운영체제에 요청이 반복되어 전체적인 응답 속도가 느려지는 문제가 예상되었음.

**[행동]** 각 처리 흐름(스레드)이 전용 전송 버퍼를 독립적으로 갖도록 설계하여, 여러 데이터 조각을 최대 1MB 단위로 모은 뒤 한 번에 전송하는 방식을 구현함. 고정 배열을 재사용하고 ArraySegment 슬라이싱을 적용해 불필요한 메모리 할당을 최소화함.

**[결과]** 운영체제 전송 요청 횟수를 대폭 줄여 네트워크 처리 부하를 감소시킴. 버퍼를 스레드별로 분리해 여러 처리 흐름이 동시에 전송해도 서로 충돌하지 않는 구조 완성.

---

### 1-2. 수신 데이터 버퍼 최적화

**[행동]** 데이터를 받을 때 메모리를 새로 할당하지 않고 고정 크기 버퍼를 재사용하는 방식으로 구현함. 읽은 위치와 쓴 위치를 별도로 관리해 데이터를 읽을 때 복사가 발생하지 않도록 처리. 버퍼가 가득 찰 때만 한 번에 정리하는 방식으로 불필요한 처리를 최소화함.

**[결과]** 모든 접속 세션에 동일하게 적용되어 접속자가 늘어날수록 메모리 절약 효과가 커짐. 메모리 정리 작업이 줄어 서버 내부 가비지 수집 부담이 감소함.

---

### 1-3. 전 구간 패킷 암호화 구현

**[상황]** 클라이언트와 서버 사이의 통신 데이터가 암호화되지 않으면 외부에서 이를 가로채 치트 도구 제작에 활용할 수 있는 보안 위협이 존재했음.

**[행동]** 업계 표준인 AES-256 암호화를 적용하여 모든 통신 데이터를 암호화함. 블록이 반복되면 패턴이 드러나는 취약한 방식(ECB)을 피하고, 보안성이 높은 CBC 방식을 선택함. 암호화 키를 개발·테스트·운영 환경별로 별도 관리하는 구조를 설계함.

**[결과]** 게임 데이터 전 구간을 암호화해 외부 도구를 통한 패킷 탈취·변조를 방어함. 환경별 키 분리 관리로 운영 키가 개발 과정에서 노출될 위험을 차단함.

---

## 2. 동시 처리 성능 최적화

### 2-1. 잠금 없는 이중 대기열 구현

**[상황]** 서버의 핵심 메시지 처리 경로에서 새 작업을 넣는 쪽과 작업을 꺼내 처리하는 쪽이 같은 목록을 동시에 접근하면 서로 기다려야 하는 병목이 발생할 수 있었음.

**[행동]** 두 개의 작업 목록을 번갈아 사용하는 방식을 설계함. 한쪽에 작업을 추가하는 동안 나머지 한쪽의 작업을 처리하고, 완료되면 역할을 교체함. 역할 교체는 CPU 수준의 원자적 연산으로 처리해 잠금 없이 동작하도록 구현함.

**[결과]** 기다리거나 반복 시도하는 과정 없이 빠르게 대기열을 교체할 수 있어, 매칭 서버 등 메시지가 많이 몰리는 구간에서도 처리 성능을 안정적으로 유지함.

---

### 2-2. 무거운 작업과 상태 변경을 분리한 2단계 처리 구조

**[상황]** 게임 상태(점수, 플레이어 위치 등)를 여러 처리 흐름이 동시에 바꾸면 데이터가 꼬일 수 있음. 반면 하나씩 순서대로 처리하면 여러 CPU 코어를 놀리게 되는 비효율이 생김.

**[행동]** 작업을 두 단계로 명확히 분리함. 1단계에서는 여러 백그라운드 스레드가 데이터 해석 등 무거운 작업을 동시에 처리하고, 2단계에서는 메인 스레드 하나만 게임 상태 변경 작업을 순서대로 처리하는 파이프라인 구조를 설계함.

**[결과]** CPU 코어를 최대한 활용하면서도 게임 데이터의 일관성을 안전하게 보장하는 구조를 완성함. 이 구조를 공용 라이브러리로 분리해 모든 서버 모듈에 재사용함.

---

### 2-3. 예약 작업 시각 정렬 타이머 구현

**[상황]** 스킬 쿨타임 종료, 세션 만료, 매칭 제한 시간 등 수천 개의 예약 작업을 정확한 순서로 실행해야 했음.

**[행동]** 실행 시각을 기준으로 자동 정렬되는 우선순위 목록 자료구조를 직접 구현함. 다음 실행 시각을 보는 데 추가 연산이 없고, 작업 등록·꺼내기가 목록 크기의 로그 비례 시간 안에 처리됨. 현재 시각보다 미래인 작업에 도달하면 즉시 처리를 멈추어 불필요한 순회를 제거함.

**[결과]** 작업 수가 수천 개로 늘어나도 처리 시간이 안정적으로 유지되며, 모든 시간 기반 게임 이벤트가 정확한 순서로 동작함.

---

## 3. 데이터 인프라 설계

### 3-1. Redis 읽기/쓰기 연결 분리 및 자동 복구 구현

**[상황]** 게임 서버에서 세션 관리, 매칭 상태, 랭킹 등 다양한 용도로 Redis를 사용하면서 읽기와 쓰기 요청이 같은 연결 묶음을 공유해 쓰기가 많아질 때 읽기 속도가 저하될 수 있었음.

**[행동]** 읽기 전용 연결 묶음과 쓰기 전용 연결 묶음을 완전히 분리해 운영함. 두 풀 모두 부하가 가장 적은 연결로 요청을 자동 분배(LeastLoaded 전략)하며, 연결 오류가 3회 누적되면 자동으로 재연결하도록 설계함. 연결 응답 속도를 지속적으로 측정해 성능 이상을 감지할 수 있는 진단 체계도 포함함.

**[결과]** 읽기와 쓰기 트래픽을 분리해 서로 영향을 주지 않는 구조를 완성함. AWS ElastiCache 운영 환경에서 일시적 네트워크 이상이 발생해도 자동 복구되어 서비스 중단 없이 운영 가능.

---

## 4. 서버 아키텍처 설계

### 4-1. 역할별 독립 서버 구조 설계

**[행동]** 하나의 큰 서버 대신 로그인·로비·매칭·게임·랭킹·중계 등 기능을 6개의 독립 서버로 나눈 구조를 설계·구현함. 공통으로 사용하는 네트워크 처리 기반 코드를 라이브러리로 분리해 모든 서버가 동일한 기반 위에서 동작하도록 표준화함. 서버를 재시작하지 않고도 설정 파일 변경이 즉시 반영되는 구조를 적용함.

**[결과]** 서버별 독립 배포 및 규모 확장이 가능한 구조를 확립함. 공용 라이브러리 분리로 새 서버를 추가할 때 반복 작업이 최소화됨. 설정 변경이 무중단으로 처리됨.

---

### 4-2. 서버 간 중앙 메시지 중계 서버 구현

**[상황]** 서버가 6개로 분리되면서 각 서버가 서로 직접 연결되면 연결 수가 기하급수적으로 늘어나고, 한 서버의 주소가 바뀔 때 여러 곳의 코드를 수정해야 하는 관리 문제가 예상되었음.

**[행동]** Bridge 서버를 중앙 허브로 설계해 모든 서버 간 메시지를 이곳에서 중계함. 메시지 종류(PacketId)에 따라 올바른 대상 서버로 전달하는 라우팅 처리를 단일 핸들러에 집중시킴. 1:1 전달, 1:다수 전체 전달 등 다양한 전달 패턴을 모두 처리함.

**[결과]** 각 서버는 Bridge 주소 하나만 알면 다른 모든 서버와 통신 가능함. 새 서버를 추가하거나 서버 주소가 바뀌어도 Bridge 한 곳만 수정하면 되는 구조 완성.

---

### 4-3. 등급 기반 매칭 시스템 구현

**[행동]** 플레이어 등급별로 독립적인 게임 방 목록을 관리하는 매칭 구조를 설계함. 2인 파티는 두 명이 반드시 같은 방에 배정되도록 별도로 처리함. 여러 처리 흐름이 동시에 접근해도 충돌이 없는 자료구조를 사용하고, 매칭 사이클마다 공정하게 방을 배정함.

**[결과]** 비슷한 실력끼리 매칭되는 구조를 구현하고, 파티원이 서로 다른 방으로 분리되는 문제를 방지함. 동접 유저가 늘어나도 매칭 처리 성능이 비례해서 확장됨.

---

### 4-4. 시즌 랭킹 시스템 구현

**[행동]** 현재 시즌과 지난 시즌 랭킹 데이터를 함께 관리하는 구조를 설계함. 날짜를 기준으로 시즌 전환을 자동으로 감지하고 처리함. 게임 결과를 받으면 빠른 저장소(Redis)와 영구 저장소(MySQL)에 동시에 반영하는 파이프라인을 구축함.

**[결과]** 시즌이 바뀌는 시점에도 서비스가 중단되지 않으며, 지난 시즌 기록 조회도 지원함. 게임 결과가 랭킹에 실시간으로 반영되는 처리 흐름 완성.
