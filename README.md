# 파트 1. 프로젝트 전체 분석 요약

> **프로젝트명**: CMM Server  
> **한 줄 요약**: 여러 명이 동시에 접속해 게임을 즐길 수 있도록 지원하는 멀티플레이어 게임 서버 시스템  
> **영상**: [CBT트레일러](https://youtu.be/4HJIC_365GU?si=unZdBOggZm8_7oyq), [플레이영상1](https://youtu.be/iQbhfPMsVP8?si=v2mMTOKRN2i8ciP9), [플레이영상2](https://youtu.be/p37VRfk0C34?si=nNbjnj7v5uOP53vG)

---

## 어떤 프로젝트인가

추격자와 생존자로 나뉘어 싸우는 멀티플레이어 게임의 서버 시스템입니다.  
로그인부터 매칭, 실제 게임 진행, 결과 저장, 랭킹 집계까지 모든 서버 측 기능을 포함합니다.  
6개의 독립 서버와 여러 공용 라이브러리로 구성되어 있으며, 개발·테스트·운영 환경을 분리해 관리합니다.

---

| 분류 | 기술 | 하는 일 |

```

       ↓

       ↓

⑥ [cmm_game] 게임 진행 (100ms마다 상태 갱신)

⑦ [cmm_rank] 결과 반영 → 랭킹 점수 업데이트
       ↓
⑧ [Redis] 접속 정보·매칭 상태 빠른 저장
```

# 파트 2. 기술적 하이라이트

---

## 1. Redis Reader/Writer Split Connection Pool
**파일**: `SBLibrary/SBDatabaseLib/Redis/SBRedisConnectionMultiplexerPool.cs`

- Reader 풀 / Writer 풀 분리 운영 (읽기·쓰기 트래픽 완전 격리)
- `ConnectionSelectionStrategy.LeastLoaded`로 연결 부하 자동 분산
- 오류 3회 누적 시 `ReconnectAsync()` 자동 재연결
- `Stopwatch`로 연결 획득 지연 시간 측정 (성능 진단)

**Why important**: 읽기/쓰기 분리로 쓰기 경합 제거. LeastLoaded 전략으로 연결 간 부하를 자동 균등 분배.

---

## 2. 패킷 배치 전송 (Batch Send)
**파일**: `SBLibrary/SBSocketPacketLib/SBTcpSendBuffer.cs`, `SBLibrary/SBSocketServerLib/Session/SBTcpSession.cs`

- `ThreadLocal<SBTcpSendBuffer>`: 스레드별 독립 버퍼 → 락 경합 완전 제거
- 최대 1MB 청크 (16 × 65536 bytes), 여러 패킷을 하나의 청크에 누적 후 단일 `SendAsync` 호출
- `Buffer.BlockCopy`로 패킷들을 청크에 병합 후 단 1회 syscall
- 수신 버퍼(`SBTcpRecvBuffer`)는 `ArraySegment` 슬라이싱으로 복사 없이 데이터 구간 노출

**Why important**: 패킷 개수만큼 발생하던 syscall 횟수를 대폭 감소시킴. 스레드별 독립 버퍼로 락 없는 확장성 확보.

---

## 3. Double-Buffered Lock-Free Task Queue
**파일**: `SBLibrary/SBCommonLib/Thread/SBThreadTaskQueue.cs`

- 큐 2개를 번갈아 사용 (`ConcurrentQueue<T>[2]`)
- `SBAtomicInt.CompareExchange()`로 현재 활성 큐 인덱스를 Atomic하게 교체 (0↔1)
- 생산자(Producer): 한쪽 큐에 무락(lock-free) 추가
- 소비자(Consumer): `GetQueue()`로 반대쪽 큐 전체를 한 번에 획득
- 블로킹 없음, 스피닝 없음 — Wait-Free에 가까운 설계

**Why important**: 서버 핵심 메시지 큐(RankMatchThread 등)에서 사용. 경합 없이 고속 메시지 처리.

---

## 4. Two-Phase Background Task Processing
**파일**: `SBLibrary/SBSocketServerLib/Thread/SBBackgroundManager.cs`

- **Phase 1** (백그라운드 스레드들): 패킷 파싱 등 CPU 집약 작업을 N개 스레드에서 병렬 처리
- **Phase 2** (메인 스레드): `ProcessTaskAfter()`로 게임 상태 변경 — 단일 스레드 보장
- Per-thread 요청 큐 → 완료 큐로 이동하는 2단계 파이프라인
- Phase 1은 무락, Phase 2는 직렬 → 경합 없이 상태 일관성 유지

**Why important**: 멀티코어를 최대한 활용하면서도 게임 상태 변경은 직렬화하는 게임 서버의 이상적인 동시성 모델.

---

## 5. Binary Heap 기반 Priority Queue Job Timer
**파일**: `SBLibrary/SBSocketServerLib/Job/JobTimer.cs`

- `IComparable<T>` 구현으로 ExecTick 기준 Min-Heap 유지
- `Push` O(log N), `Pop` O(log N), `Peek` O(1)
- `Environment.TickCount` 기반 현재 시각과 비교, 미래 잡은 즉시 break
- Singleton으로 전역 관리, 여러 서버 모듈에서 주기 태스크에 활용

**Why important**: 수천 개의 예약 잡을 효율적으로 정렬/실행. 게임 서버 특성상 타이밍이 결정론적이어야 하는데 최적 구조 제공.

---

## 6. Sliding Window Receive Buffer
**파일**: `SBLibrary/SBSocketPacketLib/SBTcpRecvBuffer.cs`

- `_readPos` / `_writePos` 두 포인터로 가용 데이터 영역 추적
- `ReadSegment`: ArraySegment 슬라이싱으로 복사 없이 읽기 구간 노출
- `Clean()`: 버퍼가 꽉 찰 때만 `Array.Copy`로 한 번에 compact — Lazy Cleanup
- 가용 공간(`FreeSize`) / 데이터 크기(`DataSize`) 속성으로 명확한 상태 관리

**Why important**: TCP 스트림 수신에서 불필요한 메모리 할당을 최소화. 모든 세션에 공통 적용되므로 세션 수에 비례해 효과가 누적됨.

---

## 7. Lock-Free AtomicInt (CAS)
**파일**: `SBLibrary/SBCommonLib/Util/SBAtomicInt.cs`

- `Interlocked.CompareExchange`로 mutex 없이 원자적 상태 전환
- `CasOn()`: 0 → 1 성공 여부 반환 (락 없는 단발 점유)
- `CompareExchange()`: 0↔1 토글 (Double-Buffered Queue의 인덱스 교체에 사용)
- JobQueue의 flush 플래그 제어에도 활용 → 락 범위 최소화

**Why important**: 핫패스(모든 패킷 처리 경로)에서 mutex 대신 CAS 사용. CPU 캐시 라인 경합 없이 동시성 제어.

---

## 8. 매칭메이킹 - Rank Grade 기반 룸 할당
**파일**: `cmm_match/MatchThread/RankMatchThread.cs`

- `ConcurrentDictionary`로 무락 유저 관리 (_users, _duoUsers, _matchRoomUsers)
- `RankMatchRoomManager.GetUsableMatchRoom(rankGrade)`: 티어별 룸 풀 관리
- 듀오 파티 별도 처리: 파티원 전체를 같은 룸에 원자적으로 배치
- 매칭 스레드가 사이클마다 대기 유저를 일괄 배정

**Why important**: 락 없는 자료구조 + 등급별 분리 매칭 = 확장 가능한 매칭 시스템. 파티 정합성 보장.

---

## 9. Hub-and-Spoke 패킷 라우터 (Bridge Server)
**파일**: `cmm_bridge/PacketHandler.cs`

- 모든 서버(Lobby, Match, Game, Rank)를 Bridge를 통해 통신
- switch 기반 PacketId 라우팅 → 컴파일 타임 타입 안전성
- 단일 진입점에서 fanout broadcast, 1:1 relay, 1:N 배포 모두 처리
- 서버간 직접 연결 없이 Bridge 경유 → 토폴로지 단순화

**Why important**: 서버 추가 시 Bridge만 수정하면 됨. 각 서버는 상대 서버의 IP/포트를 몰라도 됨 → 느슨한 결합(Loose Coupling).

---

## 10. AES-256-CBC 패킷 암호화
**파일**: `SBLibrary/SBCommonLib/Crypt/SBCrypt.cs`

- AES-256 (Rijndael, 32바이트 키 / 16바이트 IV)
- CBC 모드 + PKCS7 패딩 — ECB 모드(블록 패턴 노출) 의도적 배제
- `CryptoStream` 활용으로 스트리밍 암호화 (메모리 효율적)
- 설정 파일에서 키/IV 주입 → 환경별 키 교체 가능

**Why important**: 게임 패킷 전 구간 암호화로 패킷 스니핑/조작 방어. CBC + PKCS7 조합은 패딩 오라클 공격 방지.

---

## 보너스 — Lazy<T> Singleton + volatile stop flag

**SBSingleton.cs**: `Lazy<T>` 기반 스레드 안전 싱글턴 — 초기화 이후 완전 무락(lock-free) 접근
**SBThread.cs**: `volatile bool _shouldStop` — CPU 레지스터 캐싱 방지, 정확한 스레드 종료 신호

## 1. 네트워크 전송 성능 개선

### 1-1. 패킷 묶음 전송 시스템 설계 및 구현

**[상황]** 게임 특성상 짧은 시간 안에 수십~수백 개의 작은 데이터 조각이 발생하는데, 이를 하나씩 따로 전송하면 운영체제에 요청이 반복되어 전체적인 응답 속도가 느려지는 문제가 예상되었음.

**[행동]** 각 처리 흐름(스레드)이 전용 전송 버퍼를 독립적으로 갖도록 설계하여, 여러 데이터 조각을 최대 1MB 단위로 모은 뒤 한 번에 전송하는 방식을 구현함. 고정 배열을 재사용하고 ArraySegment 슬라이싱을 적용해 불필요한 메모리 할당을 최소화함.

**[결과]** 운영체제 전송 요청 횟수를 대폭 줄여 네트워크 처리 부하를 감소시킴. 버퍼를 스레드별로 분리해 여러 처리 흐름이 동시에 전송해도 서로 충돌하지 않는 구조 완성.

---

### 1-2. 수신 데이터 버퍼 최적화

**[행동]** 데이터를 받을 때 메모리를 새로 할당하지 않고 고정 크기 버퍼를 재사용하는 방식으로 구현함. 읽은 위치와 쓴 위치를 별도로 관리해 데이터를 읽을 때 복사가 발생하지 않도록 처리. 버퍼가 가득 찰 때만 한 번에 정리하는 방식으로 불필요한 처리를 최소화함.

**[결과]** 모든 접속 세션에 동일하게 적용되어 접속자가 늘어날수록 메모리 절약 효과가 커짐. 메모리 정리 작업이 줄어 서버 내부 가비지 수집 부담이 감소함.

---

### 1-3. 전 구간 패킷 암호화 구현

**[상황]** 클라이언트와 서버 사이의 통신 데이터가 암호화되지 않으면 외부에서 이를 가로채 치트 도구 제작에 활용할 수 있는 보안 위협이 존재했음.

**[행동]** 업계 표준인 AES-256 암호화를 적용하여 모든 통신 데이터를 암호화함. 블록이 반복되면 패턴이 드러나는 취약한 방식(ECB)을 피하고, 보안성이 높은 CBC 방식을 선택함. 암호화 키를 개발·테스트·운영 환경별로 별도 관리하는 구조를 설계함.

**[결과]** 게임 데이터 전 구간을 암호화해 외부 도구를 통한 패킷 탈취·변조를 방어함. 환경별 키 분리 관리로 운영 키가 개발 과정에서 노출될 위험을 차단함.

---

## 2. 동시 처리 성능 최적화

### 2-1. 잠금 없는 이중 대기열 구현

**[상황]** 서버의 핵심 메시지 처리 경로에서 새 작업을 넣는 쪽과 작업을 꺼내 처리하는 쪽이 같은 목록을 동시에 접근하면 서로 기다려야 하는 병목이 발생할 수 있었음.

**[행동]** 두 개의 작업 목록을 번갈아 사용하는 방식을 설계함. 한쪽에 작업을 추가하는 동안 나머지 한쪽의 작업을 처리하고, 완료되면 역할을 교체함. 역할 교체는 CPU 수준의 원자적 연산으로 처리해 잠금 없이 동작하도록 구현함.

**[결과]** 기다리거나 반복 시도하는 과정 없이 빠르게 대기열을 교체할 수 있어, 매칭 서버 등 메시지가 많이 몰리는 구간에서도 처리 성능을 안정적으로 유지함.

---

### 2-2. 무거운 작업과 상태 변경을 분리한 2단계 처리 구조

**[상황]** 게임 상태(점수, 플레이어 위치 등)를 여러 처리 흐름이 동시에 바꾸면 데이터가 꼬일 수 있음. 반면 하나씩 순서대로 처리하면 여러 CPU 코어를 놀리게 되는 비효율이 생김.

**[행동]** 작업을 두 단계로 명확히 분리함. 1단계에서는 여러 백그라운드 스레드가 데이터 해석 등 무거운 작업을 동시에 처리하고, 2단계에서는 메인 스레드 하나만 게임 상태 변경 작업을 순서대로 처리하는 파이프라인 구조를 설계함.

**[결과]** CPU 코어를 최대한 활용하면서도 게임 데이터의 일관성을 안전하게 보장하는 구조를 완성함. 이 구조를 공용 라이브러리로 분리해 모든 서버 모듈에 재사용함.

---

### 2-3. 예약 작업 시각 정렬 타이머 구현

**[상황]** 스킬 쿨타임 종료, 세션 만료, 매칭 제한 시간 등 수천 개의 예약 작업을 정확한 순서로 실행해야 했음.

**[행동]** 실행 시각을 기준으로 자동 정렬되는 우선순위 목록 자료구조를 직접 구현함. 다음 실행 시각을 보는 데 추가 연산이 없고, 작업 등록·꺼내기가 목록 크기의 로그 비례 시간 안에 처리됨. 현재 시각보다 미래인 작업에 도달하면 즉시 처리를 멈추어 불필요한 순회를 제거함.

**[결과]** 작업 수가 수천 개로 늘어나도 처리 시간이 안정적으로 유지되며, 모든 시간 기반 게임 이벤트가 정확한 순서로 동작함.

---

## 3. 데이터 인프라 설계

### 3-1. Redis 읽기/쓰기 연결 분리 및 자동 복구 구현

**[상황]** 게임 서버에서 세션 관리, 매칭 상태, 랭킹 등 다양한 용도로 Redis를 사용하면서 읽기와 쓰기 요청이 같은 연결 묶음을 공유해 쓰기가 많아질 때 읽기 속도가 저하될 수 있었음.

**[행동]** 읽기 전용 연결 묶음과 쓰기 전용 연결 묶음을 완전히 분리해 운영함. 두 풀 모두 부하가 가장 적은 연결로 요청을 자동 분배(LeastLoaded 전략)하며, 연결 오류가 3회 누적되면 자동으로 재연결하도록 설계함. 연결 응답 속도를 지속적으로 측정해 성능 이상을 감지할 수 있는 진단 체계도 포함함.

**[결과]** 읽기와 쓰기 트래픽을 분리해 서로 영향을 주지 않는 구조를 완성함. AWS ElastiCache 운영 환경에서 일시적 네트워크 이상이 발생해도 자동 복구되어 서비스 중단 없이 운영 가능.

---

## 4. 서버 아키텍처 설계

### 4-1. 역할별 독립 서버 구조 설계

**[행동]** 하나의 큰 서버 대신 로그인·로비·매칭·게임·랭킹·중계 등 기능을 6개의 독립 서버로 나눈 구조를 설계·구현함. 공통으로 사용하는 네트워크 처리 기반 코드를 라이브러리로 분리해 모든 서버가 동일한 기반 위에서 동작하도록 표준화함. 서버를 재시작하지 않고도 설정 파일 변경이 즉시 반영되는 구조를 적용함.

**[결과]** 서버별 독립 배포 및 규모 확장이 가능한 구조를 확립함. 공용 라이브러리 분리로 새 서버를 추가할 때 반복 작업이 최소화됨. 설정 변경이 무중단으로 처리됨.

---

### 4-2. 서버 간 중앙 메시지 중계 서버 구현

**[상황]** 서버가 6개로 분리되면서 각 서버가 서로 직접 연결되면 연결 수가 기하급수적으로 늘어나고, 한 서버의 주소가 바뀔 때 여러 곳의 코드를 수정해야 하는 관리 문제가 예상되었음.

**[행동]** Bridge 서버를 중앙 허브로 설계해 모든 서버 간 메시지를 이곳에서 중계함. 메시지 종류(PacketId)에 따라 올바른 대상 서버로 전달하는 라우팅 처리를 단일 핸들러에 집중시킴. 1:1 전달, 1:다수 전체 전달 등 다양한 전달 패턴을 모두 처리함.

**[결과]** 각 서버는 Bridge 주소 하나만 알면 다른 모든 서버와 통신 가능함. 새 서버를 추가하거나 서버 주소가 바뀌어도 Bridge 한 곳만 수정하면 되는 구조 완성.

---

### 4-3. 등급 기반 매칭 시스템 구현

**[행동]** 플레이어 등급별로 독립적인 게임 방 목록을 관리하는 매칭 구조를 설계함. 2인 파티는 두 명이 반드시 같은 방에 배정되도록 별도로 처리함. 여러 처리 흐름이 동시에 접근해도 충돌이 없는 자료구조를 사용하고, 매칭 사이클마다 공정하게 방을 배정함.

**[결과]** 비슷한 실력끼리 매칭되는 구조를 구현하고, 파티원이 서로 다른 방으로 분리되는 문제를 방지함. 동접 유저가 늘어나도 매칭 처리 성능이 비례해서 확장됨.

---

### 4-4. 시즌 랭킹 시스템 구현

**[행동]** 현재 시즌과 지난 시즌 랭킹 데이터를 함께 관리하는 구조를 설계함. 날짜를 기준으로 시즌 전환을 자동으로 감지하고 처리함. 게임 결과를 받으면 빠른 저장소(Redis)와 영구 저장소(MySQL)에 동시에 반영하는 파이프라인을 구축함.

**[결과]** 시즌이 바뀌는 시점에도 서비스가 중단되지 않으며, 지난 시즌 기록 조회도 지원함. 게임 결과가 랭킹에 실시간으로 반영되는 처리 흐름 완성.
